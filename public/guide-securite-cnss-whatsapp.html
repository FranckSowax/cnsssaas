Principe fondamental : Aucune donnee bancaire sensible

La plateforme ne stocke aucun numero de compte, solde, mot de passe bancaire ou donnee financiere. Elle gere uniquement : noms, numeros de telephone, historique de conversations marketing/support, et preferences de communication.

2 Authentification & controle d'acces

2.1 Authentification JWT

L'acces a la plateforme est protege par un systeme d'authentification base sur JSON Web Tokens (JWT) :

  • Tokens signes avec algorithme HS256 et cle secrete de 256 bits minimum
  • Expiration configurable (defaut : 7 jours, ajustable a 24h pour environnement haute securite)
  • Verification de l'existence et du statut actif de l'utilisateur a chaque requete
  • Tokens transmis uniquement via header HTTP Authorization (pas de cookies, pas d'URL)
  • Refresh token avec endpoint dedie et validation complete

2.2 Controle d'acces base sur les roles (RBAC)

RoleAccesRestrictions
ADMIN Acces complet a toutes les fonctionnalites Gestion des utilisateurs, templates, campagnes, configuration systeme
MANAGER Gestion des campagnes et contacts Pas d'acces a la gestion des utilisateurs ni a la configuration systeme
OPERATOR Operations quotidiennes Lecture seule sur la configuration, pas de suppression

2.3 Securite des mots de passe

  • Hachage bcrypt avec 10 rounds de salage (standard OWASP)
  • Mots de passe jamais stockes en clair ni retournes dans les reponses API
  • Comparaison securisee via bcrypt.compare() (protection timing-attack)
  • Changement de mot de passe requiert le mot de passe actuel

2.4 Protection contre les attaques par force brute

3 Protection des donnees

3.1 Classification des donnees

CategorieDonneesSensibiliteProtection
Identite Nom, prenom Moyenne Chiffrement au repos (Supabase TDE)
Contact Telephone, email Haute Chiffrement au repos, masquage dans les logs
Conversations Messages chatbot Moyenne Chiffrement au repos, retention configurable
Credentials Mots de passe Critique Hachage bcrypt irreversible
Tokens API JWT, WhatsApp token Critique Variables d'environnement, jamais en base
Donnees bancaires Aucune - Non stockees par la plateforme

3.2 Chiffrement

En transit (TLS/SSL)

  • HTTPS force sur toutes les communications (certificat SSL Railway)
  • TLS 1.2+ pour la connexion a la base de donnees Supabase
  • TLS 1.3 pour les communications avec Meta WhatsApp Cloud API
  • Aucune communication en HTTP non chiffre

Au repos

  • Supabase : Transparent Data Encryption (AES-256) sur les volumes de stockage
  • Backups automatiques chiffres quotidiens (retention 7 jours)
  • Mots de passe haches avec bcrypt (irrecuperable meme en cas d'acces a la base)
  • Cles API et tokens d'acces en variables d'environnement (jamais dans le code source)

3.3 Masquage des donnees sensibles

Les numeros de telephone sont systematiquement masques dans les logs applicatifs :

Exemple de masquage

+24107488716 → +*******8716 (seuls les 4 derniers chiffres visibles)

Les tokens, mots de passe et cles API ne sont jamais enregistres dans les logs.

4 Securite des API & communications

4.1 En-tetes de securite HTTP

Le middleware Helmet.js configure automatiquement les en-tetes de protection :

X-Download-OptionsnoopenEmpeche l'ouverture directe des telechargements
Referrer-Policyno-referrerPas de fuite de referrer

4.2 Protection contre les injections

  • SQL Injection : Prisma ORM genere des requetes parametrees (prepared statements)
  • XSS : Contenu echappe cote serveur, en-tetes Helmet configurees
  • CSRF : CORS strict limite aux origines autorisees, tokens JWT (pas de cookies session)
  • Path Traversal : Validation des chemins de fichiers avec path.resolve()
  • Command Injection : Aucune execution de commande systeme par l'application

4.3 Limitation des tailles de requete

  • Corps JSON : limite a 10 Mo
  • Upload de fichiers (CSV/Excel) : limite a 10 Mo avec validation MIME type
  • Types acceptes : text/csv, application/vnd.ms-excel, application/vnd.openxmlformats-officedocument.spreadsheetml.sheet

5 Securite WhatsApp Cloud API

5.1 Architecture de communication Meta

La plateforme communique avec l'API WhatsApp Cloud de Meta via des mecanismes de securite multi-couches :

  1. 1
    Authentification OAuth Bearer Token d'acces permanent genere depuis la console Meta Business, transmis via header HTTP Authorization.
  2. 2
    Verification Webhook HMAC-SHA256 Chaque notification entrante de Meta est signee avec un secret partage. La plateforme verifie la signature avant traitement.
  3. 3
    Chiffrement bout-en-bout WhatsApp Les messages entre l'utilisateur final et WhatsApp restent chiffres E2E (protocole Signal). La plateforme recoit le contenu dechiffre via l'API Cloud.
  4. 4
    Validation des templates Meta Tous les messages proactifs (campagnes) passent par des templates pre-approuves par Meta, empechant le spam et les contenus abusifs.

5.2 Isolation des donnees Meta

Principe de separation

Les tokens d'acces Meta (WHATSAPP_ACCESS_TOKEN, APP_SECRET) sont stockes exclusivement en variables d'environnement sur Railway. Ils ne transitent jamais par le frontend, ne sont jamais enregistres en base de donnees, et ne sont jamais inclus dans les logs.

5.3 Gestion du consentement (Opt-in / Opt-out)

  • Detection automatique des mots-cles de desinscription : STOP, ARRET, DESINSCRIRE
  • Marquage immediat du contact comme UNSUBSCRIBED
  • Exclusion automatique de toutes les campagnes futures
  • Reinscription possible via mot-cle START
  • Footer obligatoire sur chaque template marketing avec mention opt-out

6.2 Protection des donnees personnelles

Dans le contexte reglementaire gabonais et de la zone CEMAC :

  • Collecte minimale de donnees (nom, telephone, historique de communication)
  • Pas de collecte de donnees financieres, numeros de compte ou mots de passe bancaires
  • Droit de desinscription immediat et automatise
  • Hebergement des donnees en region UE (Supabase EU, Railway EU)
  • Logs applicatifs avec masquage des donnees personnelles
  • Politique de retention des donnees configurable

6.3 Audit et tracabilite

  • Journalisation complete de toutes les operations API (Morgan + Winston)
  • Horodatage de chaque action (creation, modification, suppression)
  • Suivi de la derniere connexion de chaque utilisateur
  • Historique complet des campagnes avec statut de chaque message
  • Tracabilite des webhooks WhatsApp entrants et sortants

7 Scenario A : Integration avec les bases de donnees CNSS

🗂

Connexion a une base de donnees CNSS existante

Synchronisation des contacts, segments, ou donnees client

Architecture recommandee

La plateforme ne doit jamais acceder directement a la base de donnees de production de CNSS. L'integration doit se faire via une API intermediaire (middleware) controlee par l'equipe IT CNSS.

Mesures de securite

  • API Gateway CNSS : Point d'entree unique, gere par l'equipe IT CNSS
  • Authentification mutuelle (mTLS) : Certificats client + serveur pour chaque communication
  • IP Whitelisting : Seule l'IP statique de Railway autorisee a appeler l'API CNSS
  • OAuth 2.0 Client Credentials : Token de service avec scope limite (lecture contacts, pas d'ecriture bancaire)
  • Chiffrement des donnees en transit : TLS 1.2+ obligatoire
  • Limitation des champs exposes : L'API Gateway ne retourne que nom + telephone (jamais de solde, numero de compte, etc.)
  • Rate limiting cote CNSS : L'API Gateway controle le volume de requetes
  • Journalisation bilaterale : Logs des deux cotes pour tracabilite complete

Flux de donnees type : Synchronisation contacts

  1. 1
    Demande authentifieeLa plateforme envoie une requete GET signee avec token OAuth 2.0 vers l'API Gateway CNSS
  2. 2
    Validation par le GatewayL'API Gateway verifie le token, l'IP source, le scope des permissions
  3. 3
    Requete filtreeLe Gateway interroge la base CNSS avec un SELECT limite aux champs autorises
  4. 4
    Reponse minimaleSeuls nom + telephone + segment commercial sont retournes (pas de donnees bancaires)
  5. 5
    Stockage isoleLes contacts sont stockes dans la base dediee de la plateforme WhatsApp (separee de la base bancaire)
Responsabilite partagee

L'equipe IT CNSS controle l'API Gateway et decide quelles donnees sont exposees. La plateforme WhatsApp consomme uniquement ce qui est mis a disposition. Aucun acces direct a la base bancaire n'est possible depuis la plateforme.

8 Scenario B : Notifications utilitaires (paiements, alertes)

Alertes de paiement, confirmations de virement, notifications de solde

Cas d'usage

  • Alerte de paiement recu : "Un paiement de {{montant}} FCFA a ete credite sur votre compte."
  • Rappel d'echeance : "Rappel : votre echeance de credit de {{montant}} FCFA est due le {{date}}."

Mesures de securite specifiques

  • Aucun montant ni numero de compte stocke : Les donnees transactionnelles sont transmises en temps reel et jamais persistees dans la plateforme WhatsApp
  • Templates utilitaires pre-approuves : Les templates de type UTILITY sont valides par Meta et contiennent des variables generiques ({{1}}, {{2}})
  • Webhook securise avec signature : L'API de notification CNSS appelle la plateforme via un endpoint dedie, authentifie par API Key + signature HMAC
  • File d'attente avec retry : En cas d'echec d'envoi WhatsApp, le message est re-tente automatiquement (max 3 fois)
  • Idempotence : Un identifiant unique par notification empeche les envois en double
  • Audit trail complet : Chaque notification envoyee est tracee avec : horodatage, destinataire (masque), template utilise, statut de livraison

Endpoint d'integration propose

POST /api/notifications/send

Authentification : API Key + HMAC-SHA256 signature

Payload : { "phone": "+241...", "template": "payment_confirmation", "variables": {"1": "50000 FCFA", "2": "12/02/2026"} }

Les donnees sensibles (montant, date) sont transmises comme variables, utilisees pour l'envoi immediat, puis effacees de la memoire (jamais persistees en base).

Regles strictes pour les notifications utilitaires

1. Jamais de numero de compte complet dans le message (uniquement les 4 derniers chiffres)

2. Jamais de solde total du compte

3. Les montants sont transmis comme variables temporaires, pas stockes

4. Le client doit avoir explicitement consenti (opt-in) aux notifications

9 Scenario C : Connexion au Core Banking System

🏦

Integration avec le systeme bancaire central

Consultation de solde, operations via chatbot WhatsApp

Avertissement : Niveau de securite maximal requis

Ce scenario implique l'acces a des donnees bancaires sensibles. Il necessite une validation formelle par la Direction des Risques, la DSI, et la conformite reglementaire avant toute implementation.

Architecture Zero Trust recommandee

  1. 1
    Verification d'identite renforceeAvant toute consultation bancaire via WhatsApp, le client doit s'authentifier via un OTP (One-Time Password) envoye par SMS au numero enregistre dans le Core Banking.
  2. 2
    API Gateway bancaire (DMZ)Un micro-service dedie, heberge dans le reseau interne CNSS (DMZ), sert d'intermediaire unique entre la plateforme WhatsApp et le Core Banking System. Il expose des endpoints en lecture seule.
  3. 3
    Authentification mutuelle (mTLS)Communication chiffree avec certificats mutuels entre la plateforme et l'API Gateway bancaire. Les certificats sont emis par la PKI interne CNSS.
  4. 4
    Tokenisation des donneesLes numeros de compte sont remplaces par des tokens opaques. Le mapping token ↔ compte est gere exclusivement par le Core Banking (jamais expose a la plateforme WhatsApp).
  5. 5
    Reponses ephemeresLes donnees bancaires (solde, releve) sont affichees au client puis immediatement purgees de la memoire de la plateforme. Aucun solde ni historique de transaction n'est stocke dans la base WhatsApp.
  6. 6
    Session timeout strictLa session bancaire via WhatsApp expire apres 5 minutes d'inactivite. Toute nouvelle consultation necessite une re-authentification.

11 FAQ — Questions anticipees DSI & Informaticiens

Q1 : Quelles donnees bancaires sont stockees ?

Aucune donnee bancaire.

La plateforme ne stocke ni numeros de compte, ni soldes, ni historiques de transactions, ni mots de passe bancaires. Seuls les noms, numeros de telephone et historiques de conversation marketing/support sont conserves.

Q2 : Que se passe-t-il si le token WhatsApp est compromis ?

Le token Meta est stocke en variable d'environnement securisee (pas dans le code ni en base). En cas de compromission : revocation immediate depuis la console Meta Business, generation d'un nouveau token, mise a jour de la variable d'environnement. Aucun redemarrage necessaire, la prise d'effet est immediat.

Q3 : Comment garantir que la plateforme n'accede pas aux donnees bancaires ?

Par conception architecturale (separation des responsabilites) :

  • La plateforme n'a aucune connaissance de l'existence du Core Banking System
  • En cas d'integration future, un API Gateway controle par l'equipe IT CNSS sert d'intermediaire exclusif
  • Le Gateway decide quelles donnees exposer (principe du moindre privilege)
  • Audit de chaque requete des deux cotes

Q4 : Comment gerer les notifications de paiement sans stocker les montants ?

Le flux est de type "fire and forget" :

  • Le systeme bancaire envoie une requete avec telephone + variables du template (montant, date)
  • La plateforme injecte les variables dans le template WhatsApp et envoie immediatement
  • Les variables (montant, date) sont en memoire vive uniquement, jamais ecrites en base
  • Seul le statut de livraison (envoye/delivre/lu/echoue) est persiste, sans les variables

Q5 : Quel est le risque si la base de donnees est compromise ?

Impact limite car :

  • Les mots de passe sont haches (bcrypt, irreversible)
  • Aucune donnee bancaire en base
  • Les tokens API Meta et OpenAI ne sont pas en base (variables d'environnement)
  • L'attaquant obtiendrait : noms, telephones, historique de conversations chatbot
  • Mesure d'attenuation : chiffrement des champs telephone/email prevu en Phase 3

Q6 : Comment la plateforme gere-t-elle la conformite GDPR/donnees personnelles ?

  • Collecte minimale (nom + telephone uniquement)
  • Droit de desinscription automatise (STOP par WhatsApp)
  • Suppression de contact possible par l'administrateur
  • Masquage des donnees personnelles dans les logs
  • Politique de retention des donnees configurable

Q7 : Peut-on heberger la plateforme sur les serveurs internes CNSS ?

Oui, c'est possible et recommande pour un deploiement en production avancee. La plateforme est un serveur Node.js standard qui peut etre deploye sur :

  • Un serveur Linux interne (Docker ou bare metal)
  • Un cluster Kubernetes CNSS
  • Un cloud prive (VMware, OpenStack)

Q8 : Quelle est la disponibilite et la resilience ?

  • Redemarrage automatique en cas de crash avec zero downtime
  • Replicas en lecture et backups automatiques quotidiens
  • Health check integre (/api/health) pour monitoring externe
  • Logs structures pour integration SIEM (format JSON)

Q9 : Peut-on auditer le code source ?

Oui, absolument.

Le code source complet est disponible sur le repository GitHub prive (FranckSowax/cnss-wacbt). L'equipe securite CNSS peut effectuer un audit de code, un test d'intrusion, ou une revue architecturale a tout moment. Nous encourageons cette demarche.

Q10 : Comment s'integrer avec l'Active Directory / LDAP de CNSS ?

L'integration SSO est realisable via :

  • SAML 2.0 : Integration avec ADFS (Active Directory Federation Services)
  • OIDC / OAuth 2.0 : Via Azure AD ou Keycloak
  • LDAP direct : Ajout d'un middleware passport-ldapauth en Node.js

L'implementation requiert la cooperation de l'equipe IT CNSS pour configurer le fournisseur d'identite.

Q11 : Quel est le cout de la plateforme en tokens IA ?

La plateforme integre un tableau de bord de suivi de consommation IA dans les Parametres, affichant en temps reel :

  • Nombre total de tokens consommes (prompt + completion)
  • Cout en FCFA par service (chatbot, embedding, enrichissement, rapports)
  • Historique quotidien sur 30 jours
  • Tarification transparente par modele utilise
